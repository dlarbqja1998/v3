# 골라바유 v3 신규 구축 진행 가이드

## 1. 지금 결정한 것

v3는 기존 프로젝트를 뜯어고치는 방식이 아니라 새로 만든다. 기존 v2는 운영 유지용으로 남기고, v3는 네이버 지도 기반 로컬 생활 플랫폼으로 다시 설계한다.

## 2. 왜 새로 만드는가

현재 구조는 기존 맛집 목록 서비스에 지도, 학식, 셔틀, 모임 기능이 붙은 형태다. v3의 목표는 지도 중심 생활 플랫폼이므로 장소 데이터, 지도 핀, 구역, 셔틀, 학식이 처음부터 같은 구조 안에 있어야 한다.

기존 DB를 고치면서 진행하면 다음 문제가 생긴다.

1. v2 운영 데이터와 v3 실험 데이터가 섞인다.
2. `restaurants` 중심 구조가 계속 발목을 잡는다.
3. 카카오 기반 장소 구조를 네이버 지도 기반 구조로 바꾸는 중간 상태가 길어진다.
4. Neon compute 사용량 원인을 v2와 v3로 분리해 보기 어렵다.

따라서 v3는 새 Neon 프로젝트와 새 Cloudflare 프로젝트를 쓰는 편이 낫다.

## 3. 비용 판단

현재 Neon만 결제 중이고 compute 시간이 커지고 있다. 새 v3 프로젝트를 만들면 비용은 사용량에 따라 늘 수 있지만, 정확히 2배가 된다고 단정할 수는 없다.

중요한 점은 v2와 v3의 compute 사용량을 분리해서 관찰할 수 있다는 것이다. v3가 새 프로젝트이면 쿼리 구조, 캐시, 지도 API 사용량, PostHog 이벤트 설계를 따로 최적화할 수 있다.

## 4. `.env` 처리 방침

기존 `.env` 파일은 참고용으로만 사용한다. 그대로 복사하지 않는다.

가져갈 것:

1. 필요한 환경변수 이름
2. 카카오 로그인에 필요한 설정 방식
3. R2, VAPID, PostHog, 관리자 키가 필요하다는 구조

새로 만들 것:

1. v3용 `DATABASE_URL`
2. v3용 `AUTH_URL`
3. v3 도메인 기준 카카오 redirect URI
4. 네이버 지도 API 키
5. 필요하면 v3용 PostHog 프로젝트 토큰
6. 필요하면 v3용 R2 bucket

절대 하지 않을 것:

1. `.env` 값을 문서에 적기
2. `.env`를 Git에 올리기
3. 기존 v2 운영 DB URL을 v3 개발 DB로 쓰기
4. v2와 v3가 같은 OAuth redirect 설정을 공유하게 두기

## 5. 첫 개발 흐름

1. 새 프로젝트 폴더를 만든다.
2. SvelteKit 앱을 새로 생성한다.
3. 기존 v2에서 로그인/온보딩 흐름만 참고해서 v3에 다시 구현한다.
4. Drizzle 스키마는 `places` 중심으로 새로 작성한다.
5. 네이버 지도 SDK 로더와 `MapProvider`를 만든다.
6. 지도 홈에서 `places`, `zones`, `categories`를 표시한다.
7. 셔틀 정류장과 학식 핀을 지도 위에 올린다.
8. PostHog 이벤트를 최소 단위로 붙인다.
9. 베타 배포 후 실제 학생 사용 흐름을 본다.

```txt
- 다음 셔틀
- 추천 식당
- 모임 진입
```
